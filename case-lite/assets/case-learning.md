# 测试用例风格规则

## 场景与测试点命名规范

- 命名模式：`操作对象 + 操作 + 结果/场景`（如"密码错误3次账号锁定"）
- 正向场景：描述成功路径（如"登录成功跳转首页"）
- 逆向/异常场景：描述失败条件和结果（如"验证码过期登录失败"）
- 边界值：明确写出边界（如"用户名长度64字符上限"）
- **不要**包含"验证"、"测试"等动词前缀

## 场景结构规范

- 按功能/角色/流程拆分场景，同一场景内测试点覆盖该场景的所有维度
- 场景层级：通常两层（场景 → 测试点），特殊情况可嵌套
- 每个场景至少覆盖：正常流程、异常/边界、权限/安全

## 执行步骤规范

- 编号列表（1. 2. 3.），每步只做一件事
- 操作对象和动作必须具体（字段名/按钮名/接口路径级别）
- 前置条件在步骤外用"**前置条件**"单独描述

## 预期结果规范

多层次验证体系，按实际场景选择适用的层：

- **UI层**：界面展示、提示信息、状态变化
- **交互层**：页面跳转、组件响应
- **接口层**：接口路径、HTTP状态码、关键字段值（可量化断言）
- **数据层**：数据库/缓存状态变更

每条预期结果必须可量化、可验证（"显示xxx"、"返回200"、"状态码403"而非"操作成功"）。

## 覆盖度要求

每个功能模块至少覆盖：
- 正常流程（主路径）
- 输入验证（非法值/边界值/必填项）
- 异常场景（网络错误/服务降级/并发）
- 权限/安全（越权/未登录/跨账号）
- 兼容/回归（多设备/多版本）

## 优先级标记

- **P0**：主流程、核心功能，必须通过才能发布
- **P1**：重要功能、常用分支、安全场景
- **P2**：边界值、兼容性、低频异常

## ✅ 风格一致性检查点

生成后逐项检查：

- [ ] 场景命名遵循"操作对象+操作+结果"模式
- [ ] 测试点命名具体且可测试（无"验证"/"测试"前缀）
- [ ] 测试点覆盖维度：正常/异常/边界/安全/兼容
- [ ] 执行步骤用编号列表，每步具体到字段/按钮名
- [ ] 预期结果包含多层验证，断言可量化
- [ ] 优先级标记合理（P0/P1/P2）
- [ ] 覆盖度不低于同类模块参考案例
