# 用例设计基线（通用）

## 1. 场景设计方法

### 1.1 四步拆解
1. 先画状态轴：用户态 / 数据态 / 对象态 / 环境态。
2. 按用户主链路拆一级场景：进入、浏览、操作、跳转、返回。
3. 按风险拆二级场景：边界、异常、性能感知、可观测性。
4. 保持粒度稳定：场景是业务子流程，测试点是可断言行为。

### 1.2 命名规则
- 场景：`对象 + 条件 + 目标行为`
- 测试点：`触发动作 + 预期结果`

## 2. 测试点设计方法

### 2.1 五类必查
1. 主流程正确性
2. 状态切换正确性
3. 边界与冲突
4. 异常与恢复
5. 可观测性（事件/日志/指标）

### 2.2 断言标准
- 断言必须可执行、可量化、可复现。
- 避免“正常/正确/符合预期”等空泛描述。
- 至少包含一个可核查信号：UI、接口、数据、日志、事件。

## 3. APP（C端）检查基线
- 登录与会话：未登录、已登录、会话失效、回跳恢复。
- 生命周期：冷启动、热启动、前后台切换。
- 内容流：首屏、刷新、分页、空态、异常态。
- 交互链路：跳转、返回、滚动位置保持。
- 弱网与失败：超时、重试、降级、兜底。
- 事件链路：曝光/点击/转化完整性。

## 4. WEB（B端）检查基线
- 权限矩阵：角色、页面、按钮、数据域。
- 查询系统：筛选、搜索、排序、分页、重置。
- 表单/批量：校验、联动、幂等、防重复提交。
- 一致性：列表-详情-导出一致。
- 异常治理：接口错误、部分成功、回滚提示。
- 审计追溯：操作日志和变更记录。

## 5. 输出建议
生成用例时按以下结构输出：
1. Scope Summary
2. State Matrix
3. Scene List
4. Test Points（含优先级）
5. Coverage Checklist
6. Assumptions & Risks
